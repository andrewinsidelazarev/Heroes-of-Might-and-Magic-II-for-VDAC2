# План памяти и вывода карты HMM2

Документ фиксирует базовую архитектуру вывода adventure map без превышения
`RAM_G` FT812 1 МБ и без загрязнения 4 МБ ОЗУ ZX Evolution. Это не план
разбиения карты на произвольные куски. Единица работы рендера — текущий экран
карты плюс запас по краям для плавного скролла.

## Жёсткие ограничения

- ZX Evolution RAM: 4 МБ, страницы по 16 КБ.
- FT812 `RAM_G`: 1 МБ.
- Физический режим FT812: 1024x768.
- Логический viewport игры: 640x480, nearest upscale 8/5.
- В аппаратную инициализацию FT812/VDAC2 не лезть без отдельной команды.
- Нельзя держать в `RAM_G` полный object atlas карты.
- Нельзя предсобирать display list объектов для всех положений экрана.
- Нельзя фильтровать объекты “по смыслу” как финальное решение: порядок и
  состав объектов должны идти из оригинальной карты и таблиц fheroes2/OpenHMM2.

## Роли памяти

### Core

Core хранит только:

- старт и диспетчер;
- базовый paging;
- ввод;
- минимальный render loop;
- DMA/SPI хелперы;
- текущие координаты экрана, героя, курсора;
- маленькие таблицы и указатели.

В Core нельзя класть:

- полные карты;
- полные таблицы объектов;
- большие display list;
- декодеры ICN/AGG;
- данные городов, боя, экранов интерфейса.

### Z80 RAM pages

Страницы ОЗУ хранят:

- compact map;
- passability;
- таблицу описания объектов карты;
- таблицу ссылок на ICN/index;
- страницы с готовыми/декодированными спрайтами для загрузки в FT812;
- код загрузчика текущего экрана карты;
- отдельные режимы игры: карта, город, бой, меню.

Правило: данные карты и объектов должны быть адресуемыми и документированными,
чтобы не искать потом, где что лежит.

### FT812 RAM_G

`RAM_G` — это не хранилище всей карты. Это рабочая видеопамять текущего режима.

Базовая раскладка `RAM_G` для adventure map с terrain atlas:

| Диапазон | Назначение |
| --- | --- |
| `#000000-#06FFFF` | terrain atlas текущей карты/режима |
| `#070000-#07FFFF` | резерв под служебные bitmap/table данные |
| `#080000-#0E7FFF` | object sprite cache текущего экрана |
| `#0E8000-#0EFFFF` | hero/cursor/малые overlay sprites |
| `#0F0000-#0FFFFF` | резерв под runtime/debug/будущие UI элементы |

Лимиты перед загрузкой:

- terrain atlas не должен выходить за `#070000`;
- object sprite cache не должен выходить за `#0E8000`;
- overlay sprites не должны пересекаться с object cache;
- суммарно запись в `RAM_G` никогда не должна выходить за `#100000`.

`CMD_INFLATE` FT812 разрешён для подгрузки остальных элементов экрана
из zlib-потоков, но он не меняет эти лимиты. Coprocessor распаковывает данные
в обычный `RAM_G`, а не рендерит из сжатого источника.

## Вариант фона: pseudo-DXT как в Zuma

Для карты можно пробовать не держать terrain atlas целиком в `RAM_G`, а
генерировать фон текущего экрана в pseudo-DXT формате:

- слой `c0`: RGB565, уменьшенный в 4 раза по каждой оси;
- слой `c1`: RGB565, такой же размер;
- слой `mask`: L4 на полном логическом размере scroll-buffer;
- вывод: как в Zuma — сначала L4 mask пишет alpha, затем два RGB565 endpoint
  слоя смешиваются через alpha;
- апскейл 640x480 → 1024x768 выполняется FT812 nearest 8/5, а не пересчётом
  игровой логики.

Для scroll-buffer 21x16 тайлов:

- логический размер: 672x512;
- endpoint-слои: 168x128;
- `c0`: `168 * 128 * 2 = 43008` байт;
- `c1`: `168 * 128 * 2 = 43008` байт;
- `mask L4`: `672 * 512 / 2 = 172032` байт;
- всего фон: `258048` байт.

Это меньше текущего terrain atlas `450560` байт примерно на `192512` байт.
Освобождённое место можно отдать под object cache, героев, врагов, нейтралов,
курсор и debug overlay.

Правило для pseudo-DXT фона: пересобирать и подгружать фон при смене
`ViewportOriginX/Y`, а pixel-scroll внутри 21x16 scroll-buffer делать сдвигом
FT812, как сейчас. Не грузить фон каждый пиксель скролла.

Подробный расчёт скролла L4 RAW зафиксирован в `Docs/dxt_l4_scroll.md`.
Кратко: для физического режима 1024x768 и логического окна 640x480 L4 mask
скроллится матрицей `A/E=160`, `C/F=rem*256`, а RGB565 endpoint-слои тем же
остатком, но с учётом DXT-сетки 4x4: `A/E=40`, `C/F=rem*64`. Все три прохода
обязаны получать один и тот же пиксельный остаток, иначе mask и endpoint-слои
разъедутся.

## Форматы спрайтов

Не все спрайты должны иметь один формат.

- Главный герой, герои противников, ключевые персонажи и активные actors:
  `ARGB4`. Причина: качество, прозрачность, анимация, визуальный приоритет.
- Неглавные объекты карты, декор, статические ресурсы, большинство фоновых
  object parts: `PALETTED4444`. Причина: 1 байт/пиксель вместо 2 байт/пиксель
  у `ARGB4`, при сохранении 4-битной alpha через палитру.
- Terrain/background: отдельный путь, предпочтительно pseudo-DXT для текущего
  scroll-buffer.

Следствие для `RAM_G`: object cache карты считать как `PALETTED4444`, actor
reserve считать как `ARGB4`. Не смешивать эти бюджеты.

## Единица вывода карты

Рендер работает не с произвольным разбиением карты, а с текущим экраном:

- видимая область: 20x15 тайлов;
- для плавного скролла нужен запас: 21x16 тайлов;
- при необходимости для крупных объектов загрузчик может учитывать объектные
  части, которые начинаются за экраном, но визуально заходят в экран.

Экран задаётся:

- `ViewportPixelX/Y` — точный пиксельный сдвиг;
- `ViewportOriginX/Y` — левый верхний тайл для таблицы terrain;
- текущий список object parts, которые надо вывести поверх terrain.

## Алгоритм вывода одного экрана

1. По `ViewportOriginX/Y` определить прямоугольник 21x16 тайлов.
2. Пройти тайлы в оригинальном порядке.
3. Для каждого тайла прочитать:
   - terrain index и flip flags;
   - bottom object part;
   - top object part;
   - addon chain.
4. Упорядочить object parts как в оригинале/fheroes2:
   - ground/layer части;
   - top части;
   - без ручной смысловой фильтрации.
5. Составить список уникальных `(ICN, index)` для текущего экрана.
6. Назначить каждому уникальному спрайту адрес в object sprite cache `RAM_G`.
7. Если спрайт уже в cache и данные совпадают — не грузить повторно.
8. Если спрайта нет — загрузить его по SPI в назначенный адрес.
9. Построить object display list текущего экрана:
   - `BITMAP_SOURCE` на адрес в cache;
   - `BITMAP_LAYOUT` по размеру спрайта;
   - `BITMAP_SIZE` с nearest upscale 8/5;
   - `VERTEX2F` с учётом `ViewportPixelX/Y`.
10. Отправить кадр через текущий vsync/DMA pipeline.

## SPI

SPI используется для подгрузки данных, а не для хранения всего заранее.

Правила:

- terrain грузится при входе на карту или смене набора terrain-тайлов;
- object sprites грузятся при смене видимого экрана, если их нет в cache;
- object/UI/actor источники можно хранить zlib-сжатыми и распаковывать в
  назначенный слот `RAM_G` через `CMD_INFLATE`;
- города/бой/экраны интерфейса имеют свои наборы загрузки и не должны жить в
  `RAM_G` одновременно с полной adventure map графикой;
- при нехватке object cache сначала выгружаются спрайты, которые не входят в
  текущий экран.

Для pseudo-DXT terrain-фона `CMD_INFLATE` не является основным решением
скролла: загрузчику нужен прямоугольный доступ к строкам `c0`, `c1` и `mask`.
Полный фон карты остаётся источником в SPI/SD, а в `RAM_G` обновляется только
рабочее окно 672x512.

HMM2 — пошаговая стратегия, поэтому допустима подгрузка при переходе экрана.
Главное — не трогать аппаратную инициализацию и не ломать кадровый swap.

## Инварианты

- `Init_Video.asm` не менять без прямой команды.
- Debug path overlay не удалять.
- Генераторы обязаны проверять предел `RAM_G`.
- Любой новый адресный диапазон заносить в документацию.
- Если объект не отображается из-за лимита, это ошибка загрузчика/кэша, а не
  повод фильтровать объект как “ненужный”.
- Сначала диагностический Python/эмуляторный отчёт, потом SPG.

## Следующий шаг реализации

Сделать генератор таблицы object parts для всей карты:

- tile x/y;
- screen-relative draw order key;
- ICN name/type;
- ICN index;
- original layer/top;
- sprite ox/oy/w/h;
- ссылка на источник данных.

После этого сделать загрузчик текущего экрана, который по этой таблице строит
рабочий object sprite cache и object display list без хранения полной карты в
`RAM_G`.
