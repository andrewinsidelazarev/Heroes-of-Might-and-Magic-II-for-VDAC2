# Карта архитектуры HMM2 — волновой обход графа состояний

Сгенерировано волновой симуляцией (BFS, 15 узлов) по fheroes2 mainGameLoop. План переноса по фронтам.

## Порядок волны (реализация)
- ФРОНТ 0 (исток) — РЕЗИДЕНТНЫЙ ЗАГРУЗЧИК (наш, не fheroes2). Оживить sd_zc (SD_CONF=#77, SD_DATA=#57, CMD17 single-block, BYTE addressing, sd_lba_in_range) + RawPak_* (FAT32 поверх SD, OpenRoot строит sector run-table один раз/сессию, LevelsMapLoaded-флаг). Слой: монтаж SD → поиск PAK по имени → стрим в RAM_G чанками 32 сектора (буфер slot2 → FT.WriteMem) БЕЗ чередования SD-чтения и DMA. Диагностика ZiFi_DbgGamesA (granular коды #25/#A1/#A6/#A7) резидентна в slot1 для F12-дампа. Проверка на эмуляторе (моделирует порты 0x57/0x77).
- ФРОНТ 0b — PAK-ГЕНЕРАТОР (Python, формат 'HPAK'/'HMM2', как Zuma make_level_pack.py): экстракторы из HEROES2.AGG/исходника fheroes2 пишут .PAK + *_pak_table.inc, встроить в build.cmd. Тестовый PAK (1 спрайт) → прогнать через ФРОНТ 0 → снимок эмулятора. Это gate перед любым экраном.
- ФРОНТ 1 — ГЛАВНОЕ МЕНЮ (MAIN_MENU): первый реальный экран, грузится загрузчиком из HMM2MENU.PAK (НЕ раздувать SPG; страницы PAK ≤ #F0 — предел платы). Диспетчер сцен в slot1, фон HEROES[0] раскроем на тайлы в RAM_G, кнопки BTNSHNGL, анимация фонарей SHNGANIM. Узел самозамкнут (settingsArea→openGameSettings→MAIN_MENU).
- ФРОНТ 2 — КУСТ МЕНЮ поверх того же RAM_G/PAK (общий HMM2MENU.PAK, фон HEROES[0] уже резидентен): NEW_GAME (выбор типа), LOAD_GAME (подменю + ленивый HMM2FILE.PAK), HIGHSCORES (HSBKG/MINIMON — отдельный HMM2HISC.PAK), CREDITS (постраничный HMM2CRED.PAK), NEW_STANDARD (инстант-переход, ассетов 0). Атлас кнопок BUTTONS_NEW_GAME_MENU. Все наследуют MUS::MAINMENU (не перезапуск).
- ФРОНТ 3 — ВЫБОР СЦЕНАРИЯ/ИГРОКОВ (SELECT_SCENARIO_*_HUMAN_PLAYER): HMM2SCEN.PAK (REQSBKG, REQUESTS-иконки, DIFFICULTY_ICON, NGEXTRA-полоса игроков, METALLIC_BORDERED_TEXTBOX, кнопки MAPSIZE/OKAY/CANCEL). Метаданные карт (Maps::FileInfo) стримятся заголовками .mp2 с SD, НЕ из графического PAK. Курсор сложности/расы/гандикапа — MovableSprite. Параллельная ветка: NEW_SUCCESSION_WARS_CAMPAIGN (видео CHOOSE.SMK → MJPEG-AVI или fallback статик-кадр).
- ФРОНТ 3b — BATTLE_ONLY (из меню): HMM2BTLO.PAK (SURDRBKG рамка, врезки SWAPWIN, кнопки RESET/START/EXIT, CELLWIN, GROUND32-террейн, бары героя) + стрим 2 портретов PORT_BIG по требованию. Тонкий узел NEW_BATTLE_ONLY→START_BATTLE_ONLY_MODE→Battle::Loader.
- ФРОНТ 4 — ADVENTURE-КАРТА (START_GAME) — главный игровой цикл. HMM2UI.PAK (ADVBORD-рамка раскроем, ADVBTNS 8 кнопок, STONBACK, HEROLOGO, статус-иконки, ROUTE) + потоковый HMM2MAP.PAK (TIL::GROUND32 32×32 тайлсет + OBJNxxxx/MONS32 видимой области, подгрузка по скроллу). Внутренний автомат END_TURN/HumanTurn. Музыка MUS::FromGround.
- ФРОНТ 5 — ВЛОЖЕННЫЕ ДИАЛОГИ ИЗ ADVENTURE (RAM_G переиспользуется под сцену, UI.PAK выгружается): ЗАМОК (HMM2CSTL_<RACE>.PAK по 6 расам, TOWNBKG+постройки), БОЙ (HMM2BTL_<ground>.PAK фон арены + HMM2UNITS лениво по участникам + HMM2BDLG диалоги боя), ГЕРОЙ (HMM2HERO.PAK: HEROBKG+оверлеи+бары), ОБЗОР КОРОЛЕВСТВА (HMM2KING.PAK: OVERBACK+OVERVIEW+SCROLL). Возврат всегда в START_GAME (бой/победа может уводить в HIGHSCORES).

## Граф состояний
ЛЕГЕНДА: [GM]=полноценный GameMode mainGameLoop; (dlg)=вложенный диалог-цикл, не GameMode (возврат в родителя).

LOADER(узел 0, резидент slot1) -> [MAIN_MENU]   ; загрузчик монтирует SD, грузит HMM2MENU.PAK, отдаёт управление диспетчеру

[MAIN_MENU] -> [NEW_GAME], [LOAD_GAME], [HIGHSCORES_STANDARD], [CREDITS], [QUIT_GAME], [EDITOR_MAIN_MENU(только POL)], [MAIN_MENU](self: дверь→Settings)

[NEW_GAME] -> [NEW_STANDARD], [NEW_BATTLE_ONLY], [NEW_SUCCESSION_WARS_CAMPAIGN], [NEW_PRICE_OF_LOYALTY_CAMPAIGN(только POL)], [SELECT_SCENARIO_TWO..SIX_HUMAN_PLAYERS](HotSeat), [MAIN_MENU](Cancel/Settings), [QUIT_GAME]

[NEW_STANDARD] -> [SELECT_SCENARIO_ONE_HUMAN_PLAYER]   ; headless инстант-переход, ассетов 0

[SELECT_SCENARIO_*_HUMAN_PLAYER] -> [START_GAME](OK→LoadNewMap), [MAIN_MENU](Cancel/нет карт), [QUIT_GAME]
   |-- (dlg) Dialog::SelectScenario (список карт, HMM2FILE-подобный стрим заголовков) -> возврат в SELECT_SCENARIO
   |-- (dlg) Dialog::inputString (имя игрока) -> возврат

[NEW_BATTLE_ONLY] -> [START_BATTLE_ONLY_MODE] -> (dlg)Battle::Only::setup -> Battle::Loader(бой) -> [MAIN_MENU](всегда после боя)
   setup (dlg) self: Reset; (dlg) selectHeroes/selectMonster/ArmyBar/ArtifactsBar

[NEW_SUCCESSION_WARS_CAMPAIGN] -> [SELECT_CAMPAIGN_SCENARIO](единственный исход; CHOOSE.SMK выбор Roland/Archibald)
[NEW_PRICE_OF_LOYALTY_CAMPAIGN] -> [SELECT_CAMPAIGN_SCENARIO]   ; (фронт 3+, за пределами замапленного куста)

[LOAD_GAME] -> [LOAD_STANDARD], [LOAD_CAMPAIGN], [LOAD_HOT_SEAT], [MAIN_MENU](Cancel), [LOAD_GAME](self: SelectFileLoad=Cancel)
[LOAD_STANDARD/CAMPAIGN/HOT_SEAT] -> [START_GAME](Load OK), [COMPLETE_CAMPAIGN_SCENARIO_FROM_LOAD_FILE], [LOAD_GAME](self: пусто)

[HIGHSCORES_STANDARD] <-> [HIGHSCORES_CAMPAIGN] (кнопка buttonOtherHighScore, внутренний цикл) ; -> [MAIN_MENU](Exit), [QUIT_GAME]
   |-- (dlg) Dialog::inputString 'Your Name' (после победы)

[CREDITS] -> [MAIN_MENU](любой ввод)   ; (контекст кампании: ShowCredits(false)->WIN.SMK->HIGHSCORES_CAMPAIGN)

[START_GAME] (внутри собственный автомат, while res==END_TURN):
   -> [END_TURN](внутренний, остаётся в StartGame), [NEW_GAME], [LOAD_GAME], [QUIT_GAME], [MAIN_MENU](аварийно), [HIGHSCORES_STANDARD](победа)
   |-- (dlg) Castle::OpenDialog [замок]  -- self: Next/PrevCastle; -> Construction/MageGuild/Tavern/Well/Marketplace/Recruit/ThievesGuild/BuyBoat/Hero
   |-- (dlg) Heroes::OpenDialog [герой]  -- self: PREV/NEXT hero; DISMISS->Adventure
   |-- (dlg) Kingdom::openOverviewDialog [обзор]  -- вкладки Heroes/Towns; -> OpenCastleDialog/OpenHeroesDialog
   |-- (dlg) Battle::Loader [бой]  -> Result(WINS/LOSS/RETREAT/SURRENDER)->возврат в adventure; победа сценария->[HIGHSCORES_STANDARD]
   |-- (dlg) Cast Spell / Adventure Options(View World/Puzzle/Dig) / System Options / Scenario Info

[QUIT_GAME] -> выход из mainGameLoop -> LOADER quit-stub (ProbeBootHobeta/BOOT.$C)

## PAK-план
- HMM2MENU.PAK (фронт 1-2, грузится один раз на весь куст меню, RAM_G резидентен пока в меню): фон HEROES[0] нарезанный на тайлы ≤256/319px (PALETTED8 ~307КБ, не 4444 614КБ!) + палитра меню; гонтовые подложки BTNSHNGL[1,5,9,13,17] + hover-кадры 1..19 (+20..22 POL); анимация фонарей+дверь SHNGANIM[0..18]; атлас кнопок BUTTONS_NEW_GAME_MENU_GOOD (28 спрайтов released/pressed: standard/campaign/multi/battle/settings/cancel/hotseat/2..6players/SW/PoL); рамка SURDRBKG[0] + гем-уголок REDBACK[0]. Музыка MUS::MAINMENU — отдельный аудио-стрим с SD, НЕ в RAM_G.
- HMM2MENU_EVIL.PAK (опционально, только при evil-интерфейсе): STONEBAK_EVIL, BUTTONS_NEW_GAME_MENU_EVIL, EVIL_DIALOG_PLAIN_CORNERS, WINLOSEE, SURDRBKE. Не грузить при good.
- HMM2FILE.PAK (ленивая загрузка при LOAD_STANDARD/CAMPAIGN/HOT_SEAT и при Dialog::SelectScenario — файловый браузер): SCROLL/SCROLLE (скроллбар 0-4), CELLWIN/CELLWIN_EVIL, OKAY/CANCEL, BUTTON_VIRTUAL_KEYBOARD_GOOD/EVIL. Фон тот же HEROES[0] — не дублировать. Имена/метаданные сейвов и .mp2 — ФС-данные, не графика.
- HMM2HISC.PAK (фронт 2, при входе в HIGHSCORES): HSBKG[0] фон 640×480 (раскрой ≤320, держать PALETTED8); HISCORE[6]=Standard/[7]=Campaign баннеры; MINIMON весь нужный набор монстров (по index*9 + кадры анимации 0..6); 4 кнопки BUTTON_HSCORES_VERTICAL_* (STANDARD/CAMPAIGN/EXIT, активный интерфейс); glyph-атлас normalWhite+normalYellow если шрифт не резидентный. Файл fheroes2.hgs — сейв на SD, читается парсером, НЕ в PAK.
- HMM2CRED.PAK (фронт 2, постранично, держать 1 страницу +префетч следующей): 5..8 пред-отрендеренных PALETTED4444-страниц (фон+впечённый текст+монстры, DARKENING применён ОФФЛАЙН): CBKGLAVA×2+CBKGSWMP (Resurrection), CBKGWATR×2 darkened (SW), +CBKGGRAV×3 darkened (POL). Header-полоса версии. 1 страница 640×432@4444 ~553КБ — влезает одна.
- HMM2SCEN.PAK (фронт 3, при Game::SelectScenario): REQSBKG[0]+[1]тень (раскрой ≤319), REQUESTS целиком (стрелки+иконки игроков/размеров/условий 18×18), MAP_TYPE_ICON (3), 5×BUTTON_MAPSIZE, METALLIC_BORDERED_TEXTBOX (нужный интерфейс), BUTTON_MAP_SELECT, BUTTON_SMALL_OKAY/CANCEL, 5×DIFFICULTY_ICON (77×69 раскрой), NGEXTRA целиком (полоса игроков: player-type цвета/wide+24, classIcon расы, гандикап 0/1/2/78, тени 59/60/61, курсор 62), ESCROLL[3]. Фон HEROES[0] унаследован из меню.
- HMM2BTLO.PAK (фронт 3b, NEW_BATTLE_ONLY): SURDRBKG рамка раскроем (+SURDRBKE evil), врезки SWAPWIN[0] (рамки портретов 111×105, полоса заголовка 496×19 пополам), кнопки RESET/START/EXIT (norm+press), CELLWIN[1]+[2] чекбокс, TIL::GROUND32 10 типов террейна, общий UI-атлас баров. Портреты PORT_BIG — стрим по 2 из HMM2PORT.PAK.
- HMM2UI.PAK (фронт 4, ADVENTURE, грузится в START_GAME, выгружается в замке/бою): ADVBORD+ADVBORDE рамка (раскрой на куски — оригинал уже собран из углов+сегментов), ADVBTNS+ADVEBTNS (8 кнопок×2), STONBACK+STONBAKE, HEROLOGO+HEROLOGE, RESSMALL/RESOURCE/HOURGLAS/BRCREST статус, ROUTE+ROUTERED. Может держаться резидентно в RAM_G на время карты.
- HMM2MAP.PAK (фронт 4, ПОТОКОВЫЙ, самый тяжёлый): TIL::GROUND32 полный тайлсет 32×32 + используемые на конкретной карте семейства OBJNxxxx/MONS32/мини-спрайты/туман SMALCLOD. Тайлы и объекты подгружать кусками вокруг видимой game-area (как уровни Zuma), не всю карту.
- HMM2CSTL_<RACE>.PAK ×6 (фронт 5, по расе текущего замка, 1 резидентен за раз): TOWNBKG<race>[0] фон 640×480 (главный тяжёлый, раскрой), полный набор спрайтов построек GetICNBuilding(race) с анимкадрами (флаги/мельницы/вода/дым), лодка GetICNBoat(race), extra-оверлеи TWN?EXT*. Грузить только реально построенные (built-bitmask), не весь расовый набор. Под-диалоги (CASLWIND/TOWNWIND/RECR*/marketplace) — свои лёгкие под-PAK по требованию. Музыка MUS::FromRace + M82::BUILDTWN — аудио-стрим.
- HMM2BTL_<ground>.PAK (фронт 5, по типу грунта при входе в бой): 1 фон CBKG* раскроем на 2×2≤320 + рамка FRNG* + декор COBJ*. Гекс-сетка/тени — примитивы FT812, НЕ bitmap.
- HMM2UNITS.PAK (фронт 5, ленивый при бое): спрайт-листы ТОЛЬКО монстров обеих армий (idle/move/attack/wince/death) — самый переменный кусок, кадры лениво по фазе хода. HMM2BHERO.PAK: CMBTHRO*/CMBTCAP* участвующих рас + флаги HEROFL00..06. HMM2SIEGE_<race>.PAK (только осада): CASTBKG<race>+стены+CATAPULT+MOATWHOL. HMM2BDLG.PAK (ленивый при диалоге боя): VGENBKG/E, SURDRBKG/E, WINLOSE/E+WINLOSEB+WINCMBT, VIEWGEN, ARTIFACT, CSPANEL. TEXTBAR/MONS32 — лёгкая панель, резидентно на время боя.
- HMM2HERO.PAK (фронт 5, окно героя): HEROBKG[0] раскрой на тайлы ≤256, оверлеи HEROEXTG[0]+HEROEXTE[0], HSBTNS[4..8], BUTTON_VERTICAL_EXIT/DISMISS+DISMISS_DISABLED, HSICONS (формации 9/10/11+индикаторы), CREST (6 цветов). Портрет PORT_BIG по id из HMM2PORT.PAK. Иконки артефактов/навыков — общий HMM2ICONS.PAK.
- HMM2KING.PAK (фронт 5, обзор королевства): OVERBACK[0] раскрой ≤319 (жёлтые grid-линии — рисовать примитивом RECT/LINE, не bitmap), OVERVIEW[6,7,8,10,11,13,14], SCROLL[0..4], 3 кнопки BUTTON_KINGDOM_*. Мелкие иконки строк (порты/существа/артефакты/навыки/ресурсы) — общий HMM2ICONS.PAK.
- HMM2ICONS.PAK (общий, переиспользуется Castle/Hero/Kingdom/Battle): ARTIFACT, MINISS/SECSKILL вторичные навыки, MONS32 мини-монстры, RESOURCE, PORT_SMALL, иконки баров. Кандидат держать полу-резидентно если RAM_G позволяет. HMM2PORT.PAK: PORT_BIG ~70 портретов, стрим по id.
- HMM2CHOOSE.PAK (видео-куст кампании): INTRO.SMK(одноразово)+CHOOSE.SMK(зацикл фон)+CHOOSEW.SMK(аудио) → транскод в MJPEG-AVI (эталон ftview, CMD_PLAYVIDEO+FIFO) ИЛИ fallback статик-кадр CHOOSE PALETTED4444 + 2 hit-зоны.

## Резидентное ядро (slot1)
- ДИСПЕТЧЕР СЦЕН (slot1): ветвление GameMode в Update/Render — это уже реализовано (задача #14 done). Конечный автомат mainGameLoop: текущий GameMode + таблица переходов; для каждого узла Update(hit-test/события) + Render(сборка DL).
- РЕЗИДЕНТНЫЙ ЗАГРУЗЧИК (slot1): sd_zc (CMD17 single-block, порты SD_CONF=#77/SD_DATA=#57, sd_lba_in_range) + RawPak_* (FAT32, OpenRoot+sector run-table, LevelsMapLoaded флаг) + overlay-трамплины (мапят LOADER_OVL_PAGE=#40 в slot3, DI вокруг restore, CF=результат через стек). Стример RAM_G чанками 32 сектора через буфер slot2→FT.WriteMem, БЕЗ чередования SD-чтение/DMA.
- ДИАГНОСТИКА ЗАГРУЗКИ (slot1, для F12-дампа независимо от slot3): ZiFi_DbgGamesA granular коды (#25 PAK found, #A1 BPB CMD17 err, #A2 bps!=512, #A6 PAK not found, #A7 run-table overflow) — бисекция упавшей загрузки.
- GAME STATE КОРОЛЕВСТВА (slot1, НЕ ассеты-пиксели, переживает смену сцен): структура world (тайлы/объекты как данные), Kingdom (7 ресурсов, список замков+итератор для Prev/Next, список героев+итератор), Players (до 8: цвет/раса/тип/гандикап/имя), Castle (race/color/built-bitmask/гарнизон), Heroes (ТТХ/опыт/мана/артефакты/навыки/армия), фокус (hero/castle), пути героев, туман+видимая область, текущий Maps::FileInfo, Game::difficulty.
- МЕЛКИЕ ДАННЫЕ-ТАБЛИЦЫ (slot1): координаты/хит-боксы всех панелей и кнопок (вычислены из геометрии StandardWindow, зашить константами), индексы кадров ICN, LUT (MUS::FromGround/FromRace, getMonsterByRating/Day, getNextTerrain, getRaceIcnIndex, MinimapTileColorTable), палитра радара 0xB5, таблицы хоткеев, счётчики анимации (lantern_frame, monsterAnimationFrameId), таймеры (MAIN_MENU_DELAY, MAPS_DELAY, CASTLE_AROUND_DELAY).
- PER-FRAME КАРКАС (slot1): рендер-цикл (сборка DL в RAM_DL 8192, отправка через CMD-FIFO 4096), палитра/CLUT текущей сцены, примитивы переиспользуемые (Render_Number/DrawDigit с нижним паддингом глифа +2px, DrawHexagon/DrawBorder линиями, паттерн статичная-панель→RAM_G-композит), needRedraw-логика (перерисовка только изменившихся ROI, не весь кадр на 14МГц).
- QUIT-STUB (slot1): ProbeBootHobeta/OVL_ProbeBoot (BOOT.$C) для выхода из игры в загрузчик ОС.

## Зоны RAM_G
"RAM_G FT812 = 1МБ, переиспользуемый по сценам (1 тяжёлая сцена за раз; ассеты держим в PALETTED8 ~1 байт/пиксель + общий CLUT, а НЕ PALETTED4444 2 байта/пиксель, где альфа не нужна — иначе фон 640×480 съедает 614КБ вместо 307КБ). ЗОНА A (резидентно-полупостоянная, ~небольшая): общий CLUT + общий HMM2ICONS-атлас (артефакты/навыки/монстры/ресурсы/мелкие порты) одним BITMAP_SOURCE с разными CELL — переиспользуется Castle/Hero/Kingdom/Battle, грузить один раз. ЗОНА B (СЦЕННЫЙ ПУЛ, основная масса, выгружается/перегружается при смене сцены): фон текущего экрана (раскроенный на тайлы ≤256/319px) + спрайты этого экрана. Бюджеты по фронтам: МЕНЮ HEROES[0] PALETTED8 ~307КБ + кнопки/анимация в остаток (good влезает; evil — отдельная перезаливка). HIGHSCORES HSBKG PALETTED8 ~307КБ + MINIMON-кадры + кнопки в отдельный под-пул. CREDITS — только 1 страница ~553КБ@4444 + префетч следующей во время NO_ACTION (100 тиков ~5с хватает на подкачку с SD). ADVENTURE — UI-атлас (рамка+кнопки+статус, небольшой, держим резидентно на время карты в ЗОНЕ B1) + РОТИРУЕМЫЙ буфер тайлов/объектов видимой области (ЗОНА B2, DMA-подгрузка по скроллу с запасом тайлов за краем ROI, чтобы не отставать на 14МГц). ЗАМОК — TOWNBKG<race> ~307КБ@8 + только построенные постройки (built-bitmask, не весь расовый набор); при Prev/Next на замок ДРУГОЙ расы — полная перезаливка ЗОНЫ B + смена музыки. БОЙ — узкое место: фон-плитки + спрайты ВСЕХ юнитов обеих армий + портреты; держать только активный кадр-набор, кадры монстра (idle/attack/move/death) подгружать лениво по фазе хода; фон осады отдельно. ПЕРЕХОД СЦЕН: на инстант-узлах (NEW_STANDARD) и во время NO_ACTION (CREDITS) — асинхронно начинать стрим следующего PAK в свободный регион ЗОНЫ B, пока текущий экран ещё на FT812 (перекрытие загрузки и отсутствия перерисовки). Fade/alpha (FadeBuilding, alphaHero, credits-fade) — через COLOR_A в DL, НЕ альфа-блит в RAM_G (экономит FIFO и не трогает ЗОНУ B). Тени гексов/grid-линии/радар-курсор — примитивы FT812 (LINE/RECT/полигоны), НЕ bitmap — освобождает RAM_G."

## Риски
- RAM_G 1МБ — главное узкое место по всем фронтам. Фон 640×480 в PALETTED4444 = 614КБ (2 байта/пиксель) почти весь бюджет → ОБЯЗАТЕЛЬНО PALETTED8 ~307КБ + CLUT там, где альфа не нужна (фоны HEROES/HSBKG/TOWNBKG/OVERBACK/HEROBKG). 4444 только для спрайтов с прозрачностью/тенями (NGEXTRA тени 59/60/61, иконки, кнопки).
- БОЙ (фронт 5) — худший случай RAM_G: фон-плитки + спрайты ВСЕХ юнитов обеих армий + 2 портрета героев одновременно. Митигация: лениво грузить кадры монстра по фазе хода (idle/attack/move/death), не весь спрайт-лист; фон осады — отдельная перезаливка.
- Лимит bitmap FT812 ~319px по стороне (практический раскрой; абсолют 2048): КАЖДЫЙ полноэкранный фон 640×480 НЕ влезает одним bitmap-handle → обязателен раскрой на тайлы/полосы ≤256/319 и сборка DL из нескольких BITMAP-команд (HEROES, HSBKG, TOWNBKG, OVERBACK, HEROBKG, CBKG*, ADVBORD, REQSBKG, баннеры HISCORE/OVERVIEW>595px). Риск дробных координат при ×1.6 для сеток иконок 18×18→28.8 — округлять консистентно, иначе уползёт сетка.
- ПРЕДЕЛ СТРАНИЦ ПЛАТЫ ≤#F0 + loader overlay на #40: SPG нельзя раздувать меню-страницами. Если контент сцены не помещается в отведённые страницы — стримить с SD динамически (правило компромисса: SPI в разумных пределах), а не запекать в SPG.
- ШИНА SD↔DMA: чередование SD-чтения и FT.WriteMem-DMA в одном чанке роняет шину (известный класс багов из Zuma). Жёсткий контракт стримера: чанк целиком с SD в slot2, ПОТОМ чанк целиком DMA в RAM_G. Риск рассинхрона при подгрузке тайлов adventure во время скролла — держать запас тайлов за краем ROI.
- ВИДЕО кампании (CHOOSE.SMK, фронт 3+): нативный SMK невоспроизводим на FT812. Риск недокорма FIFO при MJPEG-AVI потоке с SD (эталон ftview). Митигация/деградация: fallback на статик-кадр выбора + 2 hit-зоны если поток не тянем на 14МГц.
- Z80 14МГц — бюджет кадра 57.25Hz: полная пересборка DL каждый кадр дорога (в Zuma ZL_DrawFrame+обновление цепочек уже выходило за 1 кадр при двух цепочках). Для HMM2: перерисовывать только изменившиеся ROI (needRedraw), статичный список построек/панелей держать в DL и менять только анимкадры; анимацию воды — palette-cycle, не пересборка.
- Мелкий глиф FT812 режет низ ~2px (snapshot не ловит, видно только на железе) — статус-бары/числа ресурсов/имена/версия в credits/highscores. Паддить все мелкие глифы снизу +2px, финал проверять на реальной плате, не на эмуляторе.
- RAM_DL 8192 / CMD-FIFO 4096: число BITMAP в DL = фон-плитки + UI + до 20 монстров (highscores) или вся армия (бой). Близко к лимиту на тяжёлых экранах — батчить блиты, генерировать DL инкрементально, тени/линии примитивами а не bitmap (экономит и RAM_G и DL).
- Метаданные карт/сейвов (Maps::FileInfo, .hgs) — стрим заголовков .mp2 с SD по мере скролла списка (видно 9 строк), НЕ держать описания всех карт в ОЗУ Z80 (#A000 уже тесно: резидент дорос до ~#A04B). Риск переполнения резидента — следить за бюджетом slot1.

АПСКЕЙЛ ×1.6: вся раскладка задана логически в 640x480, физически 1024x768 NEAREST — фон рендерить уже растянутым через BITMAP_SIZE/масштаб FT812 либо хранить апскейленные плитки; все смещения (88/244/403/484/554, шаг 40, base Y 72/91) умножать на 1.6 при формировании DL-координат. АНИМАЦИЯ МОНСТРОВ: оригинал перерисовывает только ROI колонки монстров по MAPS_DELAY (≈ кадр каждые ~ms), сохраняя/восстанавливая фон через ImageRestorer — на FT812 это БЕСПЛАТНО: монстры рисуются как отдельные BITMAP поверх статичного фона в DL, фон не трогаем, ImageRestorer не нужен; меняем только индекс кадра MINIMON (index*9+1+seq) в DL раз в N кадров дисплея. RAM_G 1МБ: фон-плитки 640x480 PALETTED4444 ≈ 600КБ (×2 байта/пиксель при 4444!) — НЕ влезает вместе с остальным; держать фон в PALETTED8 (индекс+палитра, ~307КБ) или грузить фон плитками по мере отрисовки/один раз резидентно, а под MINIMON-кадры и кнопки выделить отдельный пул RAM_G. FIFO/CMD: число BITMAP в DL = плитки фона + 1 баннер + до 10 статич.монстров + 10 анимир.монстров + 2 кнопки — укладывается в лимит DL FT812, узкое место не DL, а трафик загрузки спрайтов монстров в RAM_G при входе (грузить разом из PAK, не по кадру). Текст строк (имена/сценарии): рендерить своим глифовым движком; учесть известную оговорку проекта — реальный FT812 режет низ мелкого глифа ~2px, паддить глифы снизу.

- **Музыка:** — (нет MUS:: на этом узле). Звук — аудиоканалы, встроенные в SMK: CHOOSEW.SMK / CHOOSE.SMK воспроизводятся через Mixer::Play (game_newgame.cpp:216-221). MUS::ROLAND_CAMPAIGN_SCREEN и MUS::ARCHIBALD_CAMPAIGN_SCREEN (mus.h) стартуют только на СЛЕДУЮЩЕМ узле SELECT_CAMPAIGN_SCENARIO через playCampaignMusic() (game_campaign.cpp:876-893, REWIND_AND_PLAY_INFINITE).
- **PAK:** PAK этого узла = видео-поток выбора: INTRO.SMK (заставка, одноразовая) + CHOOSE.SMK (видео-фон, зацикленный) + CHOOSEW.SMK (аудиодорожка). Это тяжёлый медиаконтент → грузить при входе в узел отдельным PAK 'campaign_choose'. Для FT812: вместо SMK — заранее транскодированный MJPEG-AVI (по эталону ftview, см. память ftview-ft812-mjpeg-reference), кадры стримятся CMD_PLAYVIDEO из FIFO. Если видео не тянем — fallback: один статичный кадр CHOOSE (PALETTED4444 раскрой) + 2 hit-зоны. Сам ICN-набор кампании (CAMPBKGG/CAMPBKGE, CAMPXTRG/E, GOOD/EVIL_CAMPAIGN_BUTTONS, CTRACK0x) НЕ сюда — он в PAK следующего узла 'campaign_scenario'.
- **Резидент vs PAK:** РЕЗИДЕНТ (логика/мелкое состояние, держать в ОЗУ Z80): диспетчер GameMode и сам обработчик узла; CampaignSaveData (campaignId, scenarioId, выбор — записывается setCurrentScenarioInfo при клике); 2 прямоугольника hit-ROI и их координаты + центрирующий roiOffset; цвет рамки-подсветки (индекс 51); таблица хоткеев CAMPAIGN_ROLAND/ARCHIBALD. PAK (тяжёлое, в RAM_G FT812, переиспользуемое): кадры видео CHOOSE/INTRO + аудио CHOOSEW. После выбора весь медиа-PAK выгружается, RAM_G отдаётся под ассеты SELECT_CAMPAIGN_SCENARIO.
- **FT812:** 1) Лимит bitmap ≤319px: логический кадр 640×480 НЕ влезает одним bitmap-хэндлом — раскрой по тайлам/полосам (≤319 по ширине), как делается для adventure-фонов. 2) ×1.6 NEAREST 640×480→1024×768: hit-ROI считать в ЛОГИЧЕСКИХ 640×480 (322×298/224×297) ДО апскейла; масштабировать в DL bitmap-трансформом, а не пересчётом координат вручную. 3) Видео = главный риск под FT812: нативный SMK не воспроизвести — либо MJPEG-AVI через CMD_PLAYVIDEO+FIFO (эталон ftview, ровный поток с SD; следить за заполнением FIFO/недокормом), либо деградация до статического кадра-выбора. 4) RAM_G 1МБ: цельный распакованный 640×480 ARGB не помещается с запасом под двойную буферизацию — стримить полосами поверх; держать одновременно только видимые тайлы. 5) Палитра PALETTED4444 для статичного fallback-кадра и рамки-подсветки; цвет рамки (orig idx 51) перелопатить в свой CLUT. 6) Звук: встроенную в SMK дорожку отделить — на FT812 это PCM в аудиобуфер/GS-канал (ср. план звука в GS), не через видео-кодек. 7) Этот узел НЕ путать с SELECT_CAMPAIGN_SCENARIO: там тяжёлые ICN (CAMPBKGG 640×480, CAMPXTRG иконки, CTRACK-треки, GOOD/EVIL_CAMPAIGN_BUTTONS) — отдельный бюджет RAM_G/PAK.
