# HMM2 VDAC2 — карта памяти и файловая структура (PAK)

Цель: HMM2 огромна, целиком в ОЗУ Z80 (64 КБ) и RAM_G FT812 (1 МБ) не влезает.
Перейти на потоковую архитектуру как Zuma (резидент + overlay + PAK с SD) **до**
дальнейшего наращивания кода. Документ — на обсуждение/утверждение.

## 1. Проблема (факты)
- Резидент Core уже #5000–#A04B (~20 КБ, занимает slot1+slot2) — упёрлись, DL-буфер
  пришлось двигать на #A200.
- Таблицы геймплея (MinimapTileColorTable 1.3 КБ, PickupList, …) и ассеты теснят бюджет.
- Впереди: замки, бои, существа, герои, несколько карт — это в разы больше.

## 2. Паттерн Zuma (референс)
- slot0 TSLib · slot1 Core-резидент (минимум) · slot3 = overlay-регион (#C000):
  UI/loader/scene-страницы мапятся по очереди (never co-resident).
- Резидент = per-frame код + trampolines + состояние. Тяжёлое спит в overlay/PAK.
- `loader_resident.asm` (тонкий резидент-хвост) + `ts-dos.asm`/`sd_zc.asm` (loader в своём
  overlay page #40) + PAK с SD.

## 3. Целевая карта памяти HMM2 (Z80 64 КБ)
> Из Zuma берём **только паттерн** (резидент + overlay-окно + PAK). Конкретные номера
> страниц и окон (slot) HMM2 назначает **свои**, под свою раскладку — ничего не обязано
> совпадать с Zuma (#40/#41/page5 и т.п. — лишь иллюстрация подхода).

| slot | адрес | назначение |
|---|---|---|
| 0 | #0000–#3FFF | TSLib (page #00) — резидент |
| 1 | #4000–#7FFF | **HMM2 Core-резидент** (page #05): main loop, game state, диспетчер сцен, trampolines, loader-резидент, FT-команды ядро |
| 2 | #8000–#BFFF | **DL-staging + рабочие буферы** (CMD buffer, pathfinding, временные); часть — текущая Scene-overlay если ядро не влезает |
| 3 | #C000–#FFFF | **Окно подкачки**: PAK-страницы (тайлы карты / спрайты замка / юниты боя) + loader overlay, мапятся по необходимости |

Цель: ужать **резидент в slot1 (16 КБ)**. Всё остальное — overlay/PAK.

## 4. Что ВСЕГДА в ОЗУ (резидент, slot1)
- Главный цикл, ввод, диспетчер сцен (Adventure/Castle/Battle), переходы.
- **Game state королевства** (мало, но нужно всегда): ресурсы (7), день/неделя/месяц,
  герои (позиция/MP/армия-сводка), список замков с владельцами, текущий режим.
- Trampolines к overlay-функциям (мапят страницу в slot3, вызывают, восстанавливают).
- Loader-резидент (ядро SD-чтения, точка входа подкачки).
- Per-frame рендер-диспетчер (общий каркас кадра).

## 5. Что подгружается из PAK
| PAK | когда грузится | содержимое |
|---|---|---|
| `map_<N>.pak` | старт/смена карты | тайлы террейна, объекты + per-origin DL, мини-карта, passability, маршрут-спрайты |
| `castle.pak` | вход в замок | фон замка, постройки, портреты существ найма, UI замка |
| `battle.pak` | начало боя | арена, фон, UI боя |
| `creatures.pak` | бой/найм | спрайты существ + анимации (по фракции — можно дробить) |
| `heroes.pak` | по необходимости | портреты/спрайты героев, окно героя |
| `ui_common.pak` | старт | общий интерфейс приключения (ADVBORD, кнопки, шрифт, ресурс-иконки) — или держать резидентно в RAM_G, т.к. нужен всегда на карте |

## 6. RAM_G (1 МБ FT812) — переиспользуется под текущую СЦЕНУ
RAM_G не вмещает всё → делится на зоны, перезаписываемые при смене сцены:
- **Постоянная зона** (низ RAM_G): общий UI приключения (рамка, кнопки, шрифт, иконки),
  палитры — пока активна карта.
- **Сценовая зона**: Adventure = composite cache + object atlas + radar; при входе в замок
  **перезаписывается** фоном замка + постройками + портретами; бой — ареной + юнитами.
- Возврат на карту → сценовая зона перезагружается из `map_<N>.pak`.

## 7. Формат PAK (предлагаемый)
- Заголовок: magic, версия, число записей.
- Каталог: на запись — `id`, `тип` (RAM_G blob / Z80 page / palette / DL), `смещение в файле`,
  `размер`, `целевой адрес` (RAM_G addr или slot3 page).
- Тело: блоки данных (выровнены под страницу/4 байта).
- Loader: читает каталог → грузит нужные записи в RAM_G (FT.WriteMem/DMA) или в страницы
  (slot3). Генерируется тем же Python-пайплайном (`build.cmd`) из AGG/исходника —
  по [[hmm2-translation-algorithm]] экстракторы пишут не в один SPG, а в PAK-набор.

## 8. Реализация PAK (на базе Zuma RawPak — проверенный стек)
Zuma уже решил ровно эту задачу на том же железе. Переиспользуем стек:
- **sd_zc.asm** — SD CMD17 single-sector read (`sd_init`, `sd_read_sector`); карта
  инициализирована хостом (WC) до старта SPG. Порт почти 1:1 (то же железо TS-Conf).
- **RawPak_*** (из ts-dos.asm) — самодостаточный FAT32-reader поверх CMD17:
  `RawPak_OpenRoot`/`OpenFile`/`ReadSectors`/`SkipB`/`Seek0`. Находит PAK по имени, читает
  секторами в страницы/RAM_G.
- **Python-генератор PAK** (как `make_level_pack.py`): наши экстракторы
  ([[hmm2-translation-algorithm]]) пишут не в один SPG, а в набор `.PAK` + `*_pak_table.inc`
  (таблица «что на какую страницу/RAM_G-адрес»). Встроить в `build.cmd`.

Файлы PAK (FAT32, в каталоге игры на SD): `HMM2MAP.PAK` (или `HMM2_<N>.PAK` на карту),
`HMM2UI.PAK` (ui_common), `HMM2CSTL.PAK`, `HMM2BTL.PAK`, `HMM2CRE.PAK`, `HMM2HER.PAK`.

## 9. Порядок миграции (не ломая рабочее)
1. **Порт sd_zc + RawPak** в HMM2 (резидент-loader): прочитать тестовый PAK с SD, проверить
   на железе/эмуляторе (mock SD).
2. **Python PAK-генератор**: сначала выгрузить в PAK уже-RAM_G-данные (object atlas,
   composite, radar, ui_common) + таблица загрузки. Текущий SPG-путь оставить рабочим как
   fallback, пока PAK-путь не подтверждён.
3. **Диспетчер сцен** в резиденте: enum режима (Adventure/Castle/Battle) + «загрузить PAK
   сцены → войти». Adventure — первая сцена.
4. **ui_common.pak** грузится при входе в Adventure; карта (map PAK) — туда же.
5. Замок/бой — новые сцены + свои PAK, БЕЗ роста резидента.

## 10. PAK-формат (минимальный, поверх FAT32-файла)
- Заголовок: magic 'HPAK', версия, число записей.
- Каталог (N записей): `id`/`тип` (1: RAM_G blob, 2: Z80 page, 3: palette, 4: DL-blob),
  `смещение(сектор+байт)`, `размер`, `назначение` (RAM_G addr 3б ИЛИ slot-page 1б).
- Тело: блоки, выровненные на 512 (сектор) для прямого CMD17→DMA.
- Loader: OpenFile(PAK) → читать каталог → по записям грузить в RAM_G (FT.WriteMem/DMA)
  или в страницы (slot-окно). Генерируется Python-пайплайном из AGG/исходника.

## Решения (утверждены пользователем)
- **Источник PAK: SD-карта** — порт `sd_zc.asm` из Zuma. Потоковая подгрузка, масштаб как оригинал.
- **UI приключения — из `ui_common.pak`** (грузится в режиме карты, выгружается в замке/бою → больше RAM_G под сцену).
- **Размер ядра — на усмотрение разработчика**: slot1 = ядро (game state + диспетчер + loader-резидент + per-frame), slot2 = рабочие буферы + текущая сцена-overlay; ужимать резидент, вынося в overlay, по мере роста. Номера страниц/окон — свои, не из Zuma.
