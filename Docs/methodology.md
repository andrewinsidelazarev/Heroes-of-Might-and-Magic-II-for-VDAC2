# Методология переноса OpenHMM2 → HMM2_VDAC2

Цель: переносить игру **поэкранно по единому конвейеру**, без ручного пиксель-тюнинга
(«2 года искать 2 пикселя» — тупик). Каждый экран/подсистема проходит одни и те же
7 шагов. Сопутствует файловой архитектуре `memory_architecture.md` (загрузчик + PAK).

## Принцип
Игра = машина состояний экранов (splash → меню → новая игра → карта → замок → бой).
Каждый экран — самодостаточный модуль: свой PAK с ассетами, свой layout-данные, свой
рендер из данных (НЕ хардкод позиций), своя state-логика. Общий код (loader, шрифт,
кнопки, game state) — резидентный, переиспользуемый.

## 7 шагов переноса одного экрана/подсистемы
1. **Локализовать в fheroes2** (`OpenHMM2/src/fheroes2/game/game_*.cpp` и т.д.):
   выделить layout (позиции/ICN элементов), данные (тексты/числа/таблицы), state-машину
   (кнопки→переходы), музыку/звук.
2. **Извлечь автоматически** (Python-экстракторы в `Source/Tools/`, НЕ хардкод):
   - layout → `.inc`/таблица (позиции из исходника; вычисляемые — формулой);
   - ассеты (ICN/спрайты/фон) из AGG → **PAK** (PALETTED, с FT812-паддингом мелких глифов);
   - тексты/числа/таблицы → `.inc`/PAK; палитры.
3. **Разнести по памяти** (`memory_architecture.md`):
   - резидент (slot1): state-машина, game state, loader, общие примитивы;
   - PAK экрана (SD → slot3/RAM_G): ассеты только этого экрана; при входе грузится, при
     выходе выгружается (RAM_G переиспользуется).
4. **Рендер data-driven**: DL строится ЦИКЛОМ по layout-данным (позиции из `.inc`/PAK),
   а не пиксель-хардкодом. Один проход — весь экран.
5. **Логика (ASM)**: state-машина экрана зеркалит fheroes2 (кнопки → следующий экран,
   ввод, переходы). Переиспользует общие примитивы.
6. **Верифицировать**: эмулятор-снимок (быстрая итерация) + **ОБЯЗАТЕЛЬНО железо/unreal**
   (snapshot ≠ железо для пиксель-точных низов/границ/скролла — урок v013).
7. **Зафиксировать переиспользуемое**: примитивы (кнопка, текст ROM/спрайт, число, фон-
   тайлинг, диалог) — в общую библиотеку, чтобы следующий экран их брал, а не дублировал.

## Переиспользуемые примитивы (строить один раз)
- `Render_Number` (числа), вывод текста, рисование кнопки (normal/pressed), фон-рамка,
  диалоговое окно, фейд-переход, загрузка PAK-блока в RAM_G/страницу, hit-test зон.
- Каждый новый экран ДОЛЖЕН их переиспользовать — это и есть «методология вместо пикселей».

## Что НЕ делать (антипаттерны этой сессии)
- Не тюнить позиции на глаз по скриншотам (см. марафон с 8-слотами/панелью).
- Не заявлять «готово» по snapshot без проверки на железе (низ глифов, scroll).
- Не наращивать резидент бесконечно (упёрлись в #A200) — тяжёлое в PAK/overlay.

## Порядок экранов (с начала игры — по рекомендации)
1. **Splash/logo** + загрузчик (SD/PAK работает, экран загрузки).
2. **Главное меню** (`game_mainmenu.cpp`): фон, кнопки New/Load/Credits/Quit (BTNSHNGL.ICN),
   музыка MUS::MAINMENU. Первый экран, проверяющий весь конвейер (PAK→рендер→ввод→переход).
3. **Новая игра / выбор сценария** (`game_newgame*.cpp`).
4. **Adventure-карта** (уже частично есть — перевести на PAK-структуру).
5. **Замок** (`castle`), **бой** (`battle`) — новые экраны по тому же конвейеру.

Начинать с (1)-(2): экран загрузки + меню — это активирует PAK-loader (sd_zc+raw_pak уже
готовы) и обкатывает методологию на простом экране, прежде чем тяжёлые карта/замок/бой.

См. [[hmm2-translation-algorithm]], [[hmm2-file-architecture-pak]], `memory_architecture.md`.
