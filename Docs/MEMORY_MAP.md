# Карта памяти HMM2 TS-Config / VDAC2

Документ является реестром занятых страниц Z80, рабочих буферов и диапазонов
FT812 RAM_G. При добавлении новой подсистемы сначала обновить этот файл, затем
код/генераторы/`spgbld_vdac2.ini`.

Кодировка: UTF-8.

## Правила

- Ядро держать маленьким: резидентный код, базовое переключение страниц,
  инициализация FT812, общие
  хелперы.
- Крупные таблицы, карты, графику, оверлеи и временные рабочие буферы держать
  в отдельных 16К страницах.
- `CMD_ADDRESS_PTR` в странице `CorePage+1` является временным scratch-буфером
  CMD/DL для FT812. Сейчас это `#AA00` (сдвинут #9800→#A700→#AA00 по мере роста резидента:
  + анимация главного меню — hover/pressed кнопок, фонарь SHNGANIM, подсветка двери).
- **Загрузчик (raw_pak) ремапит slot2** под свой буфер: любой код/стек/данные,
  что должны пережить вызов loader, держать в slot1 (#4000-#7FFF) или slot0.
  Trampolines загрузчика и стрим-цикл лежат в slot1; стек на время loader — тоже slot1.
- Новые страницы не брать "на глаз": сначала внести запись в этот файл.
- После любого расширения ядра должен проходить `ASSERT CoreEnd <= CMD_ADDRESS_PTR`.
- Все адреса изображений в `RAM_G` держать выровненными минимум по 4 байта.
- `CMD_INFLATE` FT812 использовать только как способ загрузки сжатых zlib
  потоков в заранее выделенные слоты `RAM_G`. В бюджете `RAM_G` считать
  распакованный размер, а не размер zlib-файла.

## Слоты Z80

| Слот | Адреса | Текущее назначение |
| --- | --- | --- |
| slot0 | `#0000-#3FFF` | TSLib/page0, служебный код и отображенные регистры при `FMapAddrInit` |
| slot1 | `#4000-#7FFF` | резидентное ядро, страница `#05`, вход `#5000` |
| slot2 | `#8000-#BFFF` | хвост ядра (page `#06`, `CoreEnd` дорос — < `#AA00`); буфер CMD/DL `CMD_ADDRESS_PTR=#AA00`; стек `#BFFF` вниз. ВАЖНО: raw_pak временно ремапит slot2 под dir/sector-буфер (`#A1`) → стек/код загрузчика на это время ОБЯЗАНЫ быть в slot1 (см. trampolines) |
| slot3 | `#C000-#FFFF` | переключаемое окно: PAK-loader overlay (page `#A0`), data/runtime-страницы |

## Резидентные адреса Core

| Адрес | Размер | Назначение |
| --- | ---: | --- |
| `#4000` | переменный | область IM2 / таблица векторов прерываний |
| `#41F3` | 2 | `ResolutionWidthPtr`, логическая ширина `640` |
| `#41F5` | 2 | `ResolutionHeightPtr`, логическая высота `480` |
| `#4200-#4247` | переменные | состояние игры, ввода, окна карты, героя, стоимостного поиска пути; `GameMode=#4203` (0=adv,1=town,2=combat,3=menu) |
| `#4248-#4257` | 16 | ресурсы королевства: `ResGold` (3б), `ResWood/Ore/Mercury/Sulfur/Crystal/Gems` (по 2б) |
| `#4258` | 1 | `MenuClickLatch` (LMB-latch меню) |
| `#4259-#4266` | 14 | `MenuNameBuf` — slot1-буфер имени PAK для loader (raw_pak ремапит slot2) |
| `#4267-#426A` | 4 | состояние меню по оригиналу: `MenuHoverIndex` (#FF=нет), `MenuLmbDown` (pressed-кадр), `MenuLanternIdx` (кадр анимации фонаря), `MenuDoorHover` (подсветка двери) |
| `#426B-#426F` | 5 | MIDI-плеер (SAM2695): `MusicActive`, `MusicWait` (кадров до события), `MusicPtr` (2б, позиция в потоке), `MusicStart` (2б, для зацикливания) |
| `#4278-#42FF` | ~136 | **стек загрузчика в slot1**: `LdSavedSP=#4278`, локальный стек `#4280..#42FF`, вершина `LdStackTop=#4300`. На время raw_pak SP переключается сюда (основной стек #BFFF в slot2 был бы испорчен ремапом) |
| `#4300-#435F` | 96 байт | `HeroPathXBuf`, X-точки текущего пути |
| `#4360-#43BF` | 96 байт | `HeroPathYBuf`, Y-точки текущего пути |
| `#5000` | код | `EntryPoint`; далее loader-trampolines (slot1!), main loop, сцены, FT, render, generated |
| `#AA00` | до `StackTop` | `CMD_ADDRESS_PTR` / `RUNTIME_DL_BUFFER`, общий временный буфер кадра (сдвинут #9800→#A700→#AA00: резидент дорос на диспетчер сцен + меню hover/pressed/фонарь/дверь + loader). Потолок старта буфера `#B070` (+#0F90 < #C000) |
| `#BFFF` | вниз | стек (slot2) |

Важно: `CMD_ADDRESS_PTR` используется последовательно: сначала как
`RUNTIME_DL_BUFFER` для генерации static DL и DMA-заливки в `RAM_G`, затем как
буфер CMD-потока FT812. Не хранить там таблицы, очереди, состояние игры или
данные, которые должны пережить `Render_Frame`.

| `#8C-#95` | свободны | резерв |

`#000000`. Так файловая структура: маленький SPG (ядро+adventure+loader) + PAK на SD.

## FT812 RAM_G

Базы фиксированы константами; точные верхние границы считает `check_ramg_usage.py`.

| Диапазон RAM_G | Статус | Назначение |
| --- | --- | --- |
| `#000000-#078FFF` | занято | кэш составного фона `PALETTED4444`, **2 банка** (double-buffer scroll): палитра `#000000`, тайлы с `#004000`; `COMPOSITE_BG_TILE_BASE=#004000`, `RAMG_OBJECT_BASE`/`OBJECT_ATLAS_RAMG=#079000` |
| `#079000-…` | занято | атлас динамики `ARGB4`/`PALETTED4444`: объекты, герой, ресурс-иконки/цифры, ресурс-панель, мини-карта (radar). **БЕЗ курсора** (вынесен) |
| `#0E0000-…` | занято | **ГЛОБАЛЬНЫЙ КУРСОР** `CURSOR_RAMG_BASE=#0E0000`: спрайты ADVMCO pointer/move/distance 1..8 (`ARGB4`), резидентны для ВСЕХ сцен (меню+adventure). Загрузка раз — `Cursor_GlobalUpload` из SPG-страницы `#A2` |
| `#0EFFFF` | граница | конец постоянной зоны курсора (до DL) |
| `#0F0000-#0F0BFF` | runtime | левый статический блок DL времени выполнения (`RUNTIME_DL_LEFT_RAMG`) |
| `#0F0C00-#0F0FBF` | runtime | правый статический блок DL времени выполнения |
| `#0F0FC0-#0F1EDB` | runtime | блок DL объектов времени выполнения, максимум `OBJECT_VIEW_DL_SIZE-4` |
| `#0F1EDC-#0FFFFF` | свободно | резерв FT812 RAM_G |

Зоны переиспользуются по сцене: меню-payload (фон HEROES + кнопки) грузится с SD из
`HMM2MENU.PAK` в `#000000` (там, где потом composite); курсор `#0E0000` и DL `#0F0000`
— ПОСТОЯННЫЕ (не перезаписываются сменой сцены).

FT812 RAM_G физически 1 МБ. Не выходить за `#0FFFFF`: адреса выше алиасятся
по младшим 20 битам и портят низ RAM_G.

`Source/Tools/check_ramg_usage.py` является обязательной проверкой сборки и
запускается из `build.cmd`. Он для каждого из `374` положений окна читает реальный
`ObjectViewDL_Table`, декодирует `BITMAP_SOURCE`, `PALETTE_SOURCE`,
`BITMAP_LAYOUT` и `CELL`, считает диапазоны чтения/записи `RAM_G` с учетом
смещения `CELL` внутри источника изображения, а также итоговый размер кадра в `RAM_DL`
FT812. Падает при:

- любом диапазоне `> #0FFFFF`;
- пересечении атласов фона/динамики с DL времени выполнения;
- источнике изображения, палитры или `CELL` вне атласа динамики;
- DL объектов, пересекающем границу 16К-страницы;
- итоговом display list больше `8192` байт.

Последняя сводка Python-конвертера без ASM build:

```text
origins=374 (17x22)
terrain atlas: #000000-#057FFF (360448 bytes)
object atlas: #058000-#05F359 (29530 bytes)
terrain/composite pages=82
object atlas pages=2
object-view pages=8
composite upload table: page #C4, 8214 bytes
```

`CMD_INFLATE` не делает bitmap "сжатым в рендере": FT812 распаковывает zlib в
обычный буфер `RAM_G`, из которого потом читает bitmap. Поэтому для объектов,
UI и спрайтов допускается хранить источник в SPI/SD/SPG сжатым, но конечный
адрес в таблице выше должен быть выделен под полный распакованный размер.
Terrain pseudo-DXT/DXT больше не является штатным путём карты приключений. Фон
текущей сборки живёт как атлас фона в `RAM_G`; DXT-генераторы оставлены
только как неактивная исследовательская ветка и не должны попадать в
дефолтную сборку.

## Внешний SPI-пак объектов

С 2026-06-12 сборка генерирует внешний пак для будущей подгрузки слоя объектов:

| Файл | Статус | Назначение |
| --- | --- | --- |
| `Assets/Converted/ObjectPacks/SKIRMISH_OBJECTS_SPI.pack` | артефакт сборки | паки объектов для всех положений окна |
| `Assets/Converted/ObjectPacks/SKIRMISH_OBJECTS_SPI.json` | артефакт сборки | манифест и аудит пака |

Формат pack:

- заголовок `H2OP`;
- таблица `374` записей для положений окна `17x22`;
- каждая запись: заголовок `H2OB`, палитра `PALETTED4444` 512 байт,
  спрайты текущего окна, готовый DL объектов текущего окна;
- второстепенные спрайты объектов: `PALETTED4444`, прозрачный индекс палитры `0`;
- план назначения для будущего runtime-загрузчика должен быть пересчитан под
  текущую разметку `RAM_G`, а не под старые адреса `#070000/#0E0FC0`.

Важно: этот пак пока НЕ является активной runtime-загрузкой. Текущая сборка
использует SPG-страницы атласа динамики и готовых DL объектов. Следующий шаг —
заменить SPG-источник этих данных на загрузчик, который при смене положения
окна читает нужную запись из SPI/SD и заливает кэш объектов плюс DL объектов
в актуальные области `RAM_G`.

## Видеорежим и координаты

| Параметр | Значение |
| --- | --- |
| физический FT812 режим | `1024x768` |
| логическое окно карты и ввод | `640x480` |
| upscale | nearest `8/5` |
| `ResolutionWidthPtr/HeightPtr` | всегда `640/480` |

Не пересчитывать gameplay/input в `1024x768`. Масштабирование выполняется при
выводе через FT812.

## Проверки

Обязательный минимум после изменения памяти:

```bat
.\build.cmd
python Source\Tools\test_hero_command.py
python Source\Tools\test_cursor_input.py
python Source\Tools\test_viewport_scroll.py
python Source\Tools\check_ramg_usage.py
python Source\Tools\hmm2_ft812_snapshot.py --out Diagnostics\hmm2_ft812_snapshot.png
```

Ожидаемые инварианты:

```text
CoreEnd <= CMD_ADDRESS_PTR
физический режим: hsize=1024 vsize=768
теневая модель FT812: swaps=1
DL: DISPLAY=1, VERTEX2II=0
калькулятор RAM_G: origins=374, max_end <= #0FFFFF
```
